# v0.136.0

Stable release of the Android Termux fork, published as
`@mmmbuto/codex-cli-termux@0.136.0` on npm `latest`.

Syncs the fork to upstream OpenAI Codex `rust-v0.136.0` and restores real
code-mode on the native Android build. Validated on a Termux arm64 device
(AI-guided suite, including the `exec`/`wait` code-mode path).

## Upstream base

- OpenAI Codex `rust-v0.136.0`
- Merge path: `rust-v0.135.0` → `rust-v0.136.0`

## Highlights

- **code-mode (`exec`/`wait`) now runs for real on Android** via the in-process
  V8 runtime (previously a no-op stub on this fork). This is the meaningful
  capability gain for Termux. It uses the fork-owned `aarch64-linux-android`
  `rusty_v8` prebuild.

## Termux fork changes (carry-forward, intact)

- Android ARM64 native package for Termux; `codex` and `codex-exec` packaged together.
- Browser login routed through `termux-open-url`.
- Fork self-update points to `DioNanos/codex-termux` and `@mmmbuto/codex-cli-termux`.
- Packaged launchers preserve `CODEX_SELF_EXE` and bundled `libc++_shared.so` visibility.
- Android ELFs use `RUNPATH=$ORIGIN` for direct native invocation.
- `flock` ENOTSUP/EOPNOTSUPP tolerance across the file-lock callsite guards.
- `TERMUX_ENV_VARS` allowlist for `npx`-spawned MCP servers.
- Maintainer builds use fork-owned Android `rusty_v8` artifacts.

## Known limitation — realtime audio on Termux

Realtime voice/audio builds for Android (cpal links `c++_shared` via
`oboe-shared-stdcxx`), but it is **not usable in a plain Termux CLI**: the audio
backend (cpal AAudio → `ndk-context`) needs an Android `JavaVM`/`Activity` that a
command-line process does not have, so opening a device panics. The experimental
`/realtime` and `/settings` commands are off by default; do not enable them on
Termux. The audio backend is intentionally left on the upstream path; a
Termux-native backend (PulseAudio / `termux-api`) is tracked separately and not
part of this release.
