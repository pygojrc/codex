# v0.133.1

This is a patch release on top of `v0.133.0` for the Android Termux fork,
published as `@mmmbuto/codex-cli-termux@0.133.1`.

## Fix

- `codex remote-control` now tolerates filesystems that reject `flock(2)`
  with `ENOTSUP` / `EOPNOTSUPP`. On Android Termux storage rooted at
  `/data/data/com.termux/files/...`, the previous build aborted daemon
  startup with the cryptic error

      Error: foreground app-server exited before remote control became ready
      Caused by:
          lock() not supported

  before the app-server could bind its socket. The two `try_lock_file`
  helpers in `app-server-daemon` (operation lock + pid reservation) now
  mirror the existing permissive degradation in
  `core::installation_id::is_unsupported_file_lock_error`: when the OS
  reports the lock primitive is not supported, the helpers return
  "lock acquired" so the start path proceeds. The exclusion these locks
  guard against is best-effort on such storage backends — refusing to
  start the daemon was the blocker, not the absent lock.

- Linux and other Unix filesystems that fully support `flock` continue to
  enforce the lock exactly as before; this change only loosens the
  error tail. Windows is unaffected.
